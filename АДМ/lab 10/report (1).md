# Цель работы

Приобретение практических навыков по конфигурированию SMTP-сервера в части настройки аутентификации.

# Выполнение

## Настройка LMTP в Dovecot

1. На виртуальной машине **server** выполнен вход под пользовательской учётной записью.  
   Для выполнения административных операций осуществлён переход в режим суперпользователя с использованием `sudo -i`.

2. В отдельном терминальном окне запущен мониторинг журнала работы почтовых служб с целью анализа доставки сообщений.  
   Использовался файл журнала `/var/log/maillog`.

3. В конфигурационном файле Dovecot `/etc/dovecot/dovecot.conf` в список поддерживаемых протоколов добавлен протокол **LMTP**.  
   Параметр `protocols` приведён к значению `imap pop3 lmtp`.

   ![Добавление протокола LMTP в dovecot.conf](Screenshot_1.png){ #fig:009 width=80% }

4. Для обеспечения взаимодействия Dovecot и Postfix по протоколу LMTP в файле  
   `/etc/dovecot/conf.d/10-master.conf` переопределён сервис **lmtp**.  
   В конфигурации указан unix-сокет `/var/spool/postfix/private/dovecot-lmtp`, а также заданы:
   - группа `postfix`,
   - пользователь `postfix`,
   - права доступа `0600`.

   Данная настройка обеспечивает корректную передачу почтовых сообщений от Postfix к Dovecot через локальный сокет.

   ![Настройка сервиса LMTP в 10-master.conf](Screenshot_2.png){ #fig:010 width=80% }

5. В Postfix выполнено переопределение способа доставки сообщений в почтовые ящики пользователей.  
   Передача писем настроена через LMTP с использованием unix-сокета Dovecot.

6. В файле `/etc/dovecot/conf.d/10-auth.conf` задан формат имени пользователя для аутентификации без указания доменной части.  
   Использован формат имени пользователя в нижнем регистре.

   ![Настройка формата имени пользователя](Screenshot_3.png){ #fig:011 width=80% }

7. Для применения внесённых изменений выполнена перезагрузка служб **Postfix** и **Dovecot**.

8. С клиентской машины под учётной записью пользователя выполнена отправка тестового письма с темой *«LMTP test»* на адрес пользователя почтового сервера.

   При мониторинге журнала `/var/log/maillog` зафиксирована следующая последовательность событий:
   - приём сообщения службой Postfix;
   - передача письма через транспорт LMTP;
   - приём сообщения службой Dovecot;
   - сохранение письма в почтовый ящик пользователя;
   - удаление сообщения из очереди Postfix.

   ![Фрагмент журнала maillog при доставке LMTP](Screenshot_4.png){ #fig:012 width=80% }

9. На сервере выполнен просмотр содержимого почтового ящика пользователя, использующего формат **Maildir**.  
   В списке писем присутствует сообщение с темой *«LMTP test»*, что подтверждает успешную доставку почты.

   ![Просмотр почтового ящика Maildir](Screenshot_5.png){ #fig:013 width=80% }

## Настройка SMTP-аутентификации

1. В файле `/etc/dovecot/conf.d/10-master.conf` определена служба аутентификации пользователей **auth**.  
   Конфигурация включает:
   - unix-сокет `/var/spool/postfix/private/auth` для взаимодействия с Postfix;
   - указание пользователя и группы `postfix`;
   - задание прав доступа `0660`;
   - дополнительный сокет `auth-userdb` для доступа Dovecot к базе пользователей, работающий от имени пользователя `dovecot` с правами `0600`.

   ![Определение службы auth в Dovecot](Screenshot_6.png){ #fig:014 width=80% }

2. Для Postfix задан тип SASL-аутентификации **dovecot** и указан путь к соответствующему unix-сокету аутентификации.

   ![Настройка SASL-аутентификации в Postfix](Screenshot_7.png){ #fig:015 width=80% }

3. В Postfix настроены ограничения на приём почты с целью предотвращения использования сервера в качестве SMTP relay.  
   Использованы следующие параметры:
   - запрет доставки писем для неизвестных доменов получателей;
   - разрешение приёма почты от доверенных сетей;
   - запрет некорректных имён получателей;
   - запрет неавторизованной пересылки;
   - разрешение доставки корректных сообщений.

4. Приём почты ограничен локальной сетью сервера путём указания адресного диапазона `127.0.0.0/8`.

5. Для тестирования SMTP-аутентификации в файле `/etc/postfix/master.cf` временно изменена конфигурация службы **smtp** с включением SASL-аутентификации и ослабленными ограничениями для авторизованных пользователей.

   ![Изменение master.cf для тестирования SMTP AUTH](Screenshot_8.png){ #fig:016 width=80% }

6. После изменения конфигурации выполнена перезагрузка служб **Postfix** и **Dovecot**.

7. На клиентской машине установлен пакет **telnet**, после чего выполнено подключение к SMTP-серверу по порту 25.

8. Для аутентификации сформирована строка в формате **Base64**, содержащая логин и пароль пользователя.

9. В ходе SMTP-сессии выполнены следующие действия:
   - проверка соединения командой `EHLO test`, подтвердившая поддержку SMTP AUTH;
   - аутентификация с использованием механизма `AUTH PLAIN`, завершившаяся успешно.

   ![Проверка SMTP-аутентификации через telnet](Screenshot_9.png){ #fig:017 width=80% }

   После завершения проверки SMTP-сессия была корректно закрыта.

## Настройка SMTP over TLS

1. На сервере выполнена настройка TLS с использованием временного сертификата Dovecot.  
   Для корректной работы с SELinux сертификат и закрытый ключ были предварительно скопированы из каталога `/etc/pki/dovecot` в стандартные каталоги TLS `/etc/pki/tls/certs` и `/etc/pki/tls/private`.

   Далее в конфигурации Postfix указаны пути к сертификату и ключу, задано расположение базы кэша TLS-сессий, а также установлен уровень безопасности TLS как *may*, что разрешает использование шифрования при наличии поддержки со стороны клиента.

   ![Настройка TLS в Postfix с использованием сертификата Dovecot](Screenshot_10.png){ #fig:018 width=80% }

2. Для организации приёма почты по протоколу SMTP over TLS выполнена модификация файла `/etc/postfix/master.cf`.  
   Основной SMTP-сервис оставлен без изменений, после чего добавлен сервис **submission**, работающий на порту 587.  
   Для него включено обязательное шифрование TLS, разрешена SASL-аутентификация и заданы ограничения на приём почты только от аутентифицированных пользователей.

   ![Настройка сервиса submission в master.cf](Screenshot_11.png){ #fig:019 width=80% }

3. В межсетевом экране сервера разрешена работа службы **smtp-submission**, обеспечивающей приём почты на 587-м порту.  
   После добавления сервиса конфигурация межсетевого экрана была перезагружена.

4. Для применения всех изменений выполнена перезагрузка службы Postfix.

5. На клиентской машине выполнено подключение к SMTP-серверу по порту 587 с использованием STARTTLS.  
   В ходе тестирования подтверждено:
   - успешное установление защищённого TLS-соединения;
   - корректный ответ сервера на команду `EHLO`;
   - возможность аутентификации пользователя с использованием механизма `AUTH PLAIN`.

   ![Проверка SMTP over TLS и аутентификации](Screenshot_12.png){ #fig:020 width=80% }

6. Дополнительно выполнена проверка отправки почтовых сообщений с использованием почтового клиента **Evolution**.  
   В настройках учётной записи для SMTP-сервера были указаны:
   - порт 587;
   - метод шифрования STARTTLS;
   - тип аутентификации — обычный пароль;
   - имя пользователя почтового сервера.

   ![Настройка SMTP over TLS в Evolution](Screenshot_13.png){ #fig:021 width=80% }

7. В интерфейсе почтового клиента зафиксировано успешное получение тестовых сообщений, отправленных через SMTP over TLS, что подтверждает корректную работу настроенного почтового сервера.

   ![Получение почтовых сообщений в Evolution](Screenshot_14.png){ #fig:022 width=80% }

8. В журнале `/var/log/maillog` сервера отражена последовательность событий, соответствующая защищённой отправке почты:
   - установка TLS-сессии;
   - успешная SASL-аутентификация пользователя;
   - передача сообщения через LMTP;
   - сохранение письма в почтовом ящике пользователя;
   - корректное завершение SMTP-сеанса.

   ![Фрагмент журнала maillog при SMTP over TLS](Screenshot_15.png){ #fig:023 width=80% }

   Это обеспечивает возможность автоматического развёртывания полностью настроенного почтового сервера при повторном запуске виртуальной машины.

   - настройку межсетевого экрана;

Реализована доставка почты с использованием протокола LMTP, настроена SMTP-аутентификация пользователей и обеспечена защищённая передача данных по протоколу SMTP over TLS.

# Контрольные вопросы

1. Приведите пример задания формата аутентификации пользователя в Dovecot в форме логина с указанием домена  
   Для аутентификации пользователя в Dovecot в форме логина с указанием домена может использоваться параметр `auth_username_format` со значением `%Lu`.  
   В этом случае имя пользователя обрабатывается в полном виде, включая доменную часть, например `user@example.net`, и приводится к нижнему регистру.

2. Какие функции выполняет почтовый Relay-сервер  
   Почтовый Relay-сервер выполняет функции промежуточного узла при передаче электронной почты между почтовыми серверами.  
   Он принимает сообщения от клиентов или других серверов и пересылает их на следующий почтовый сервер по маршруту доставки.  
   Relay-сервер может выполнять проверку подлинности отправителя, фильтрацию сообщений, антивирусную и антиспам-обработку, а также обеспечивать маршрутизацию почты между доменами.

3. Какие угрозы безопасности могут возникнуть в случае настройки почтового сервера как Relay-сервера  
   При некорректной настройке почтового Relay-сервера существует риск превращения его в открытый SMTP relay.  
   Это может привести к использованию сервера для массовых спам-рассылок, что повлечёт блокировку IP-адреса в чёрных списках.  
   Также возможны перегрузка сервера, утечка конфиденциальной информации, распространение вредоносного ПО и компрометация репутации домена.
