## Разбиение IPv6-сети на подсети

### Анализ сети 2001:db8:c0de::/48 и разбиение на 2 подсети двумя способами

1. Для заданной IPv6-сети 2001:db8:c0de::/48 определены основные характеристики:
   - тип адреса: глобальный одноадресный (Global Unicast), так как принадлежит диапазону 2000::/3
   - префикс: /48
   - маска префикса: первые 48 бит фиксированы (первые 3 хекстета)
   - адрес сети (начальный): 2001:db8:c0de:0000:0000:0000:0000:0000
   - крайние значения адресов в пределах /48:
     - минимальный адрес: 2001:db8:c0de:0000:0000:0000:0000:0000
     - максимальный адрес: 2001:db8:c0de:ffff:ffff:ffff:ffff:ffff

2. Выполнено разбиение сети на 2 подсети способом с использованием идентификатора подсети.
   В IPv6 при типовом планировании адресации граница подсети задаётся на /64, а роль идентификатора подсети обычно выполняет 4-й хекстет.
   Для получения двух подсетей из /48 выделен 1 дополнительный бит в поле идентификатора подсети, то есть выбран новый префикс /49.

   Подсеть 1:
   - сеть: 2001:db8:c0de:0000::/49
   - диапазон адресов (краевые значения):
     - минимальный: 2001:db8:c0de:0000:0000:0000:0000:0000
     - максимальный: 2001:db8:c0de:7fff:ffff:ffff:ffff:ffff

   Подсеть 2:
   - сеть: 2001:db8:c0de:8000::/49
   - диапазон адресов (краевые значения):
     - минимальный: 2001:db8:c0de:8000:0000:0000:0000:0000
     - максимальный: 2001:db8:c0de:ffff:ffff:ffff:ffff:ffff

3. Выполнено разбиение сети на 2 подсети способом с использованием идентификатора интерфейса.
   В IPv6 идентификатор интерфейса обычно занимает последние 64 бита адреса (host-часть при /64).  
   Для разбиения на 2 части выделен 1 бит из области идентификатора интерфейса, то есть префикс увеличен на 1: /49 → /49 не подходит, так как изменится именно subnet-part.  
   Поэтому фиксируется прежняя граница подсети /48, а разделение выполняется внутри каждой /64 по старшему биту интерфейсной части.

   Вариант интерпретации:
   - подсеть по-прежнему рассматривается как 2001:db8:c0de::/48,
   - но адресное пространство узлов делится на две группы по старшему биту идентификатора интерфейса (бит 64):
     - группа A: адреса с идентификатором интерфейса в диапазоне 0000:0000:0000:0000 – 7fff:ffff:ffff:ffff
     - группа B: адреса с идентификатором интерфейса в диапазоне 8000:0000:0000:0000 – ffff:ffff:ffff:ffff

   Пояснение:
   - такой подход не создаёт новые “подсети” на уровне маршрутизации, так как префикс сети остаётся /48,
   - но позволяет логически разделить конечные устройства на две группы внутри одного префикса (например, для политики адресации, ACL, DHCPv6-пулов или схемы распределения адресов).

4. Сделан вывод по вариантам разбиения:
   - разбиение по идентификатору подсети меняет префикс сети (например, /48 → /49) и формирует две независимые подсети, которые могут маршрутизироваться отдельно;
   - разбиение по идентификатору интерфейса не меняет маршрутизируемую границу префикса, но делит host-часть на диапазоны, что подходит для внутренней логики распределения адресов, без образования отдельных маршрутизируемых подсетей.


### Анализ сети 2a02:6b8::/64 и разбиение на 2 подсети двумя способами

5. Для заданной IPv6-сети 2a02:6b8::/64 определены основные характеристики:
   - тип адреса: глобальный одноадресный (Global Unicast), диапазон 2000::/3
   - префикс: /64
   - маска префикса: первые 64 бита фиксированы (первые 4 хекстета)
   - адрес сети (начальный): 2a02:06b8:0000:0000:0000:0000:0000:0000
   - крайние значения адресов в пределах /64:
     - минимальный адрес: 2a02:6b8:0:0:0000:0000:0000:0000
     - максимальный адрес: 2a02:6b8:0:0:ffff:ffff:ffff:ffff

6. Выполнено разбиение сети 2a02:6b8::/64 на 2 подсети способом с использованием идентификатора подсети.
   Для получения двух подсетей необходимо увеличить префикс на 1 бит: /64 → /65.
   В этом случае разделение происходит по старшему биту идентификатора интерфейса (так как /64 уже фиксирует subnet-part полностью),
   однако формально это является разбиением сети на две маршрутизируемые части с разными префиксами /65.

   Подсеть 1:
   - сеть: 2a02:6b8::/65
   - диапазон адресов:
     - минимальный: 2a02:6b8:0:0:0000:0000:0000:0000
     - максимальный: 2a02:6b8:0:0:7fff:ffff:ffff:ffff

   Подсеть 2:
   - сеть: 2a02:6b8:0:0:8000::/65
   - диапазон адресов:
     - минимальный: 2a02:6b8:0:0:8000:0000:0000:0000
     - максимальный: 2a02:6b8:0:0:ffff:ffff:ffff:ffff

7. Выполнено разбиение сети 2a02:6b8::/64 способом с использованием идентификатора интерфейса.
   Так как исходный префикс уже /64, вся оставшаяся часть — это идентификатор интерфейса (64 бита).
   Разделение выполнено по старшему биту идентификатора интерфейса (бит 64):
   - группа A: идентификатор интерфейса 0000:0000:0000:0000 – 7fff:ffff:ffff:ffff
   - группа B: идентификатор интерфейса 8000:0000:0000:0000 – ffff:ffff:ffff:ffff

   Пояснение:
   - это разделение можно использовать для логической группировки адресов (например, разные пулы SLAAC/DHCPv6 или разные классы устройств),
   - при сохранении общего маршрутизируемого префикса /64.

8. Сделан вывод по вариантам разбиения:
   - при разбиении “по идентификатору подсети” для /64 фактически происходит переход к префиксам /65, что является нетипичным для IPv6 LAN, но математически корректным;
   - разбиение “по идентификатору интерфейса” позволяет разделять адреса узлов внутри /64 без изменения границ маршрутизации.

   В ARP-пакетах зафиксированы запросы и ответы, используемые для сопоставления IPv4-адресов с MAC-адресами сетевых интерфейсов.  
   По данным кадрам можно определить:

   Наличие ARP-запросов подтверждает корректную работу механизма разрешения адресов в IPv4-сети.

   ![Захват ICMP-трафика IPv4](Screenshot_18.png){ #fig:018 width=80% }

   В ICMP-пакетах IPv4 наблюдаются эхо-запросы и эхо-ответы, а также сообщения о недоступности назначения.  
   Анализ ICMP позволяет получить следующую информацию:
   - факт успешной или неуспешной доставки пакетов между узлами;
   - IP-адреса источника и назначения;
   - значения TTL, по которым можно косвенно судить о количестве пройденных маршрутизаторов;
   - сообщения Destination Unreachable, указывающие на невозможность доставки пакета в определённую подсеть или к определённому сервису.

   ![Захват ICMPv6-трафика](Screenshot_19.png){ #fig:019 width=80% }

   В ICMPv6-пакетах зафиксированы эхо-запросы и эхо-ответы между IPv6-узлами.  
   По перехваченным ICMPv6-пакетам можно определить:
   - IPv6-адреса источника и назначения;
   - корректность маршрутизации IPv6-трафика;
   - значение Hop Limit, аналогичное TTL в IPv4;
   - подтверждение работоспособности IPv6-связности между узлами.
